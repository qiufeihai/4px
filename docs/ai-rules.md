## AI 协作开发规则与约束

本文件用于约束 AI 在本仓库内的开发行为，目标是减少不必要改动，并保证代理系统的高性能与高稳定性。

### 必用技能

- 在编写、修改、重构任何代码前，必须先启用 `karpathy-guidelines` 并遵循其原则（先思考、最小改动、可验证目标、避免过度工程化）。

### 性能与稳定性优先级

任何改动都必须以“代理体验不退化”为硬约束，优先级固定为：
- 低延时
- 低抖动
- 高带宽
- 高稳定

如果某个改动会让上述任一维度变差，应拒绝该改动或改为更保守的实现方式。

### 代码改动前的检查清单

在修改前必须明确回答（写在实现说明/PR 描述/对话总结里）：
- 改动是否触达 `/proxy` 数据面热路径（Node server）或 tun2socks 数据面（移动端）？
- 是否增加了每请求的额外 CPU/内存/系统调用/网络往返？
- 是否引入了额外的定时器、轮询或后台任务？如有，是否可降低频率或改为手动触发？
- 是否改变了鉴权/设备限制/准入逻辑？是否仍保留 `/proxy` 作为最终准入点？

### 变更实现原则

- 只做必要改动：每一行变更都必须能追溯到明确需求。
- 避免“为了可配置而可配置”：除非明确提出需要，否则不新增可变策略。
- 控制面与数据面分离：状态查询/心跳等应避免进入 `/proxy` 转发链路。
- 默认不引入高频日志：日志应限频、采样或只在诊断模式开启。
- 若进行了功能删减/移除：必须同步删除或更新所有相关文档描述，避免文档与实际行为不一致。
- 若新增/修改了配置字段：必须检查并同步更新所有配置文件与示例（如 `*.example.json`、`*.local.json`、模板、README），避免出现“代码已变更但配置漏改”。

### 验证与交付

每次提交前（或在对话中交付前）至少完成：
- 语法/类型检查通过
- 若改动涉及 `apps/node` 服务端代码：先执行 `cd apps/node && npm run lint`，确保 `no-undef` 等静态错误在部署前被拦截
- 关键路径自检：没有把控制面逻辑塞进 `/proxy` 热路径
- 若改动涉及性能关键路径：提供最小可验证的对比口径（例如相同场景下的延迟/抖动主观体验，或同配置的压测对比）

### 版本号与 CHANGELOG（server/client 双版本）

规则：
- 仅在你要求时，或在准备推送/发布前，才执行版本号分配与 CHANGELOG 补充。
- 版本号与 CHANGELOG 的具体流程与格式，以 [versioning.md](file:///Users/qiufeihai/github/4px/docs/versioning.md) 为准。
